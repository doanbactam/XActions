Candidate 1: Message Contract Module

26+ string-typed messages scattered across 5 files — no single source of truth

Strong

Files

  • background/service-worker.js (26 types, switch statement)
  • src/popup/lib/useAgent.ts (14 types, inline)
  • src/popup/App.tsx (3 types, inline)
  • content/bridge.js (13 types, two switches)
  • content/injected.js (6 types, inline)

Problem

The popup, dashboard, service worker, bridge, and injected script all communicate via chrome.runtime.sendMessage and window.postMessage with string type fields. There is no module that owns this contract. Each file hardcodes its own string literals. A typo in one file produces a silent { error: 'Unknown message type' } at runtime — no compile-time check, no autocomplete, no documentation of the full protocol.

Understanding "what messages can the UI send?" requires reading all 5 files and cross-referencing. This is zero locality — the knowledge is smeared across the entire codebase.

BEFORE — No module, string soup

useAgent.ts
sendMessage({ type: 'AGENT_GET_CONFIG' })
sendMessage({ type: 'XAI_OAUTH_START' })
sendMessage({ type: 'AGENT_RUN_STRATEGY' })
...14 string literals
service-worker.js
case 'AGENT_GET_CONFIG': ...
case 'XAI_OAUTH_START': ...
...26 string literals
bridge.js
case 'RUN_AUTOMATION': ...
case 'AGENT_TOOL': ...
...13 string literals

No single interface. A rename means find-and-replace across 5 files. A new message means updating 3+ files by hand.

AFTER — Deep MessageContract module

MessageContract (small interface)
// one file, one source of truth
type Msg = { type: 'AGENT_GET_CONFIG' } | ...
type Resp = { success: boolean, ... } | ...
send(msg: Msg): Promise<Resp>
handle(handler: (msg) => Resp)
↓ leverage ↓
useAgent.ts
send({ type: 'AGENT_GET_CONFIG' }) // typed
service-worker.js
handle(msg => match(msg)) // exhaustive
bridge.js
forward(msg) // typed relay

One interface, one source of truth. TypeScript exhaustive checks catch missing handlers. Tests exercise the contract directly.

Solution

Create a single MessageContract module — a TypeScript file that defines the discriminated union of all message types and their response shapes. The module exposes two functions: send(msg) for callers and handle(dispatcher) for the service worker. The bridge gets a forward(msg) that relays typed messages across the content-script seam.

The service worker's 26-case switch becomes an exhaustive match — TypeScript flags any unhandled case at compile time. The popup and dashboard import the same types, so a rename is a single-file change.

Benefits

  • Locality: The message protocol lives in one file. Adding, renaming, or removing a message is a single edit — no cross-file find-and-replace.
  • Leverage: All 5 consumers (popup, dashboard, SW, bridge, injected) get typed send/receive for free. No more silent typos.
  • Testability: The contract is the test surface — write a test that sends each message type and asserts the response shape. No Chrome APIs needed for contract tests.
  • Deletion test: Delete the module → 26 string literals reappear across 5 files, each independently typed. Complexity concentrates. The module earns its keep.
Note: This candidate does not require an ADR — no existing decision contradicts it. It's pure addition.

Candidate 2: Service Worker Decomposition

1230-line monolith mixing 6 concerns behind one giant switch

Strong

Files

  • background/service-worker.js — 1230 lines, 1 file

Problem

The service worker is the codebase's largest module and its interface is the entire switch statement — 26 cases spanning 6 unrelated concerns: agent orchestration, OAuth flow, automation lifecycle, tab navigation, state sync, and rate-limit detection. Understanding "how does OAuth work?" requires scrolling past automation lifecycle code. Understanding "how does runPageTool retry?" requires reading through 40 lines of navigation logic embedded in the same file.

The module is not shallow — it's the opposite: too much implementation behind one unstructured interface. But the seam is wrong. There's one seam (the message listener) for 6 independent subsystems. Each subsystem's bugs, tests, and changes are forced to co-locate with 5 unrelated subsystems.

BEFORE — One monolith, one seam

service-worker.js (1230 lines)
┌─ handleMessage (26 cases) ─┐
│ Agent orchestration │
│ OAuth flow │
│ Automation lifecycle │
│ Tab navigation │
│ State sync + badge │
│ Rate limit detection │
│ Context menus │
│ Alarms health check │
└─────────────────────────────┘

All concerns share one file,
one state object, one closure.

Changing OAuth means touching the same file as automation lifecycle. No concern can be tested in isolation.

AFTER — Sub-modules behind thin router

service-worker.js (router)
handleMessage → dispatch to sub-modules
↓ seams ↓
AgentOrchestrator — runChat, runStrategy, executePlaybook
OauthFlow — start, poll, pending, logout, panel
AutomationLifecycle — start, stop, stopAll, pause, resume
TabNav — ensureXTab, navigateXTab, runPageTool
StateStore — syncState, badge, logActivity

Each sub-module owns its concern. Tests target one sub-module. The router is ~30 lines.

Solution

Keep the service worker as a thin message router (~30 lines) that dispatches to sub-modules. Each sub-module is a plain JS file with a small interface: AgentOrchestrator (chat, strategy, playbook execution), OauthFlow (device login lifecycle), AutomationLifecycle (start/stop/pause), TabNav (ensureXTab, navigate, page tool relay), StateStore (state sync, badge, activity log).

The sub-modules share a ctx object (the current state + Chrome APIs) passed by the router — no globalThis, no implicit coupling. Each can be tested by passing a fake ctx.

Benefits

  • Locality: OAuth bugs live in oauth-flow.js. Automation bugs live in automation-lifecycle.js. No more scrolling past 800 lines of unrelated code.
  • Leverage: Each sub-module's interface is small (3-5 functions). The router just dispatches — adding a new concern means adding a new sub-module, not growing the switch.
  • Testability: Each sub-module accepts a ctx with Chrome APIs injected. Tests pass a fake ctx — no Chrome environment needed.
  • Deletion test: Delete the router → 26 cases collapse back into one file. Delete a sub-module → its concern's complexity reappears in the router. Both earn their keep.
Constraint: MV3 service workers are loaded via importScripts() — sub-modules must use that pattern, not ES module imports. This is a Chrome platform constraint, not an architectural choice.

Candidate 3: Storage Schema Module

15+ chrome.storage.local keys scattered, no schema, no validation, no migration

Worth exploring

Files

  • background/service-worker.js — 6 keys (automations, activityLog, globalPaused, totalActions, firstRun, rateLimited)
  • agent/strategist.js — 4 keys (agentPlaybook, agentAccountAnalysis, agentStrategyProgress, agentStrategyLastResult)
  • agent/xai-oauth.js — 2 keys (xaiOauth, xaiOauthPending)
  • agent/tools.js — 2 keys (agentChatHistory, agentSafety)
  • src/popup/lib/useExtensionState.ts — reads 3 keys
  • service-worker.js — dynamic keys (settings_{automationId})

Problem

chrome.storage.local is the shared state layer between the service worker, popup, dashboard, and agent modules. But there is no module that owns the storage schema. Each file calls chrome.storage.local.get('someKey') with a hardcoded string. If strategist.js renames agentPlaybook to playbook, the popup silently breaks — no compile error, no runtime warning, just empty state.

The popup's useExtensionState polls 3 keys; the service worker syncs 3 different keys; the agent modules each own 2-4 keys. There's no map of the full storage layout. A new developer has to grep for chrome.storage.local across 8 files to understand what's persisted.

BEFORE — No owner, strings everywhere

strategist.js
chrome.storage.local.get('agentPlaybook')
chrome.storage.local.set({ agentPlaybook: ... })
service-worker.js
chrome.storage.local.get(['automations', 'activityLog'])
chrome.storage.local.set({ automations: ... })
xai-oauth.js
chrome.storage.local.get('xaiOauth')
chrome.storage.local.set({ xaiOauthPending: ... })

15+ string keys, no types, no validation. A typo = silent empty state.

AFTER — Deep Store module

Store (small interface)
get<T>(key: StorageKey): Promise<T>
set<T>(key: StorageKey, val: T): Promise<void>
// StorageKey = 'agent.playbook' | 'agent.config' | ...
// typed, validated, namespaced
↓ leverage ↓
strategist.js: Store.get('agent.playbook')
service-worker.js: Store.set('automation.state', ...)
xai-oauth.js: Store.get('oauth.tokens')

One typed interface. Renames are safe. Tests inject a fake Store.

Solution

Create a Store module that owns the full storage schema. It defines a StorageKey type (a union of all valid keys, namespaced like 'agent.playbook', 'oauth.tokens'), and exposes typed get / set / remove functions. All chrome.storage.local calls go through this module.

The module can also handle migrations (versioned schema) and defaults (return a default if key is missing). The popup's useExtensionState becomes Store.subscribe('automation.state', callback) — one typed subscription instead of polling 3 raw keys.

Benefits

  • Locality: The storage schema lives in one file. Adding a new key, renaming, or migrating is a single edit.
  • Leverage: All 8 consumers get typed access. No more chrome.storage.local.get('agentPlaybook') with a string that might be wrong.
  • Testability: Tests inject a fake Store (in-memory Map). No Chrome APIs needed.
  • Deletion test: Delete the module → 15+ string keys reappear across 8 files. Complexity concentrates. The module earns its keep.

Candidate 4: Shared UI Library

Dashboard imports from popup/lib — cross-boundary dependency

Worth exploring

Files

  • src/dashboard/App.tsx → imports from ../popup/lib/useAgent
  • src/dashboard/App.tsx → imports from ../popup/lib/useExtensionState
  • src/dashboard/components/ConfigSection.tsx → imports from ../../popup/lib/format
  • src/dashboard/components/ConfigSection.tsx → imports from ../../popup/lib/rpc
  • src/dashboard/components/PersonaSection.tsx → imports from ../../popup/lib/format
  • src/dashboard/components/ActivitySection.tsx → imports from ../../popup/lib/format

Problem

The dashboard was added in the most recent commit and reuses the popup's hooks (useAgent, useExtensionState) and utilities (format, rpc) by importing from ../popup/lib/. This creates a dependency where the dashboard's stability depends on the popup's internal file structure. If the popup is refactored (e.g. useAgent split into smaller hooks), the dashboard breaks.

The seam is in the wrong place. popup/lib/ is the popup's private implementation directory, but it's being used as a shared library. The popup and dashboard are sibling UIs — neither should own the other's dependencies.

BEFORE — Dashboard reaches into popup's internals

src/popup/lib/
useAgent.ts · useExtensionState.ts · rpc.ts · format.ts
↑↑↑ dashboard reaches up ↑↑↑
src/dashboard/
App.tsx → ../popup/lib/useAgent
ConfigSection → ../../popup/lib/format

Dashboard depends on popup's internal structure. Popup refactor breaks dashboard.

AFTER — Shared lib at the right seam

src/shared/lib/
useAgent.ts · useExtensionState.ts · rpc.ts · format.ts · types.ts
↓ clean imports ↓
src/popup/
imports from shared/lib
src/dashboard/
imports from shared/lib

Both UIs are siblings. Shared code lives at their common root.

Solution

Move useAgent.ts, useExtensionState.ts, rpc.ts, format.ts, and types.ts from src/popup/lib/ to src/shared/lib/. Both popup and dashboard import from ../shared/lib/. The popup's components/ directory stays popup-specific; the dashboard's components/ stays dashboard-specific.

This is a pure file move — no logic changes. The tsconfig.json already includes all of src/, so no config changes needed. The esbuild entry points already resolve relative imports.

Benefits

  • Locality: Shared hooks and types live in a location that signals "shared" — not hidden inside the popup's directory.
  • Leverage: A future third UI (e.g. sidepanel React rewrite) can import from shared/lib/ without reaching into the popup.
  • Testability: Shared hooks can be tested independently of either UI.
  • Deletion test: Delete shared/lib/ → the hooks must go somewhere. Putting them back in popup/lib/ recreates the cross-boundary dependency. The shared location earns its keep.

Candidate 5: rpc.ts — Deepen or Delete

8-line pass-through wrapper — genuinely shallow

Speculative

Files

  • src/popup/lib/rpc.ts — 8 lines, 1 function

Problem

rpc.ts is a 1-line wrapper around chrome.runtime.sendMessage with a generic type parameter. Its interface (one function, one parameter, one return) is nearly as complex as its implementation (one function call). The deletion test is clear: delete it, and each caller writes chrome.runtime.sendMessage(msg) as Promise<T> directly — complexity does not concentrate, it just moves.

However, the module has latent depth potential. Right now it does nothing with errors, timeouts, or retries. The service worker can be asleep (MV3 lifecycle) and sendMessage can fail silently. A deeper rpc.ts could handle wake-up retries, timeout, and error normalization — all hidden behind the same send() interface.

BEFORE — Shallow pass-through

rpc.ts (8 lines)
export async function sendMessage<T>(msg): Promise<T> {
  return chrome.runtime.sendMessage(msg) as Promise<T>;
}

Interface ≈ implementation. No error handling, no timeout, no retry. If SW is asleep, silent failure.

AFTER (option A) — Deepened with retry + timeout

rpc.ts (deep)
export async function send<T>(msg): Promise<T> {
  // 1. try sendMessage
  // 2. on failure, wake SW via action
  // 3. retry once
  // 4. timeout after 10s
  // 5. normalize error shape
}

Same interface to callers. 5 concerns hidden behind send(). Real depth.

AFTER (option B) — Delete

Callers use chrome.runtime.sendMessage(msg) directly. The MessageContract module (Candidate 1) provides the typing. No wrapper needed.

Solution

Two options, pick one:
A) Deepen: Add MV3 wake-up retry, timeout, and error normalization. The interface stays send(msg): Promise<resp> but the implementation handles the real-world failure modes of Chrome's service worker lifecycle. This is the right call if the team has seen silent sendMessage failures.
B) Delete: If Candidate 1 (MessageContract) is implemented, it provides typed send() directly. rpc.ts becomes redundant — delete it and let the MessageContract own the send function.

Benefits

  • If deepened: Locality — all MV3 messaging failure modes handled in one place. Leverage — every caller gets retry + timeout for free.
  • If deleted: One fewer file. The MessageContract absorbs the typing. Simpler is better if the failure modes aren't real.
  • Deletion test (current): Delete now → complexity moves, doesn't concentrate. It's not earning its keep yet.
Recommendation: If Candidate 1 (MessageContract) is adopted, delete rpc.ts — the contract module absorbs its job. If Candidate 1 is not adopted, deepen rpc.ts to handle MV3 lifecycle failures.

Top Recommendation

Start with Candidate 1: Message Contract Module

The message contract is the seam between every part of the extension — UI ↔ background ↔ content script ↔ page. Right now that seam is 50+ string literals scattered across 5 files with no owner. Every other candidate touches this seam:

  • Candidate 2 (SW decomposition) needs the contract to know which sub-module handles which message.
  • Candidate 3 (Storage schema) needs the contract for storage-related messages (AGENT_GET_CONFIG, AGENT_SAVE_CONFIG).
  • Candidate 4 (Shared lib) moves rpc.ts — which is the send half of the contract.
  • Candidate 5 (rpc.ts) is directly absorbed by the contract module.

The contract module is also the lowest-risk change: it's pure addition (a new TypeScript file), doesn't change runtime behavior, and can be adopted incrementally — one file at a time replaces string literals with typed calls. No big-bang refactor required.

Why not SW decomposition first?

The service worker split (Candidate 2) is higher impact but higher risk — it restructures 1230 lines of working code. Doing it without a typed message contract means the new sub-modules still use string literals, and a typo during the split produces a silent runtime bug.

Suggested order

  1. Message Contract (Candidate 1)
  2. Shared UI Library (Candidate 4) — trivial move
  3. Storage Schema (Candidate 3)
  4. SW Decomposition (Candidate 2)
  5. rpc.ts (Candidate 5) — delete, absorbed by #1
